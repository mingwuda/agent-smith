# Coding Cycle（自动编辑-测试-修复循环）

## 核心原则

每次修改代码后，**不需要等待用户说"跑测试"**，主动执行完整的测试循环：

```
修改代码 → 自动跑测试 → 通过 → 报告完成
                        ↘ 失败 → 读错误日志 → 修复 → 再跑测试
                                           ↘ 失败 → 再修 → 最多 3 次
                                                      ↘ 超限 → 报告失败原因
```

## 流程说明

### 阶段 1：修改代码

按常规方式修改代码（`write_file` / `edit_file` / `run_shell` 等）。

### 阶段 2：自动触发测试

修改完成后，**立即自动执行**以下操作：

1. 检测项目类型，确定测试命令：

   | 项目特征 | 测试命令 | 构建命令 |
   |----------|----------|---------|
   | `package.json`（前端/Node） | 从 `scripts` 中提取 `test` / `test:unit` / `test:e2e`，优先运行与被修改文件关联的测试 | `npm run build`（如需要） |
   | `pyproject.toml` / `setup.py` | `python -m pytest -xvs 关联测试文件` | `python -m build`（如需要） |
   | `Cargo.toml` | `cargo test` | `cargo build` |
   | `go.mod` | `go test ./...` | `go build ./...` |
   | 无识别 | `run_shell("python -m pytest -xvs 2>/dev/null || npm test 2>/dev/null || go test ./... 2>/dev/null")` | 跳过 |

2. 用 `run_shell` 执行测试命令

### 阶段 3：结果处理

#### 测试通过

```
✅ 所有测试通过，无需进一步修改。
向用户简要汇报：改了什么 + 验证结果。
继续等待下一个用户指令。
```

#### 测试失败

```
❌ 测试失败。

1. 用 run_shell 重新跑测试（加上 --verbose 获取详细输出）
2. 仔细阅读错误消息和理解失败原因
3. 分析是：
   - 代码逻辑错误 → 修复代码
   - 测试用例本身有问题 → 修正测试
   - 缺少依赖/环境问题 → 修复环境
4. 修复后重新跑测试
5. 最多重试 3 次
6. 3 次后仍失败 → 向用户汇报完整错误日志和尝试过的修复方案，请求用户协助
```

### 阶段 4：范围扩大（可选）

如果单元测试通过，且项目有端到端测试或集成测试：

```
1. 运行 npm run test:e2e 或类似命令
2. 检查浏览器截图（如有 browser_tools）
3. E2E 失败时同样进入修复循环
```

### 阶段 5：完成报告

测试全部通过后，向用户汇报：

```
✅ 开发完成
  - 修改内容：(摘要)
  - 测试结果：全部通过（单元测试 N 个 / 集成测试 N 个）
  - 覆盖率：(如有)
  - 截图：(如有 visual 变更)
```

## 关键规则

1. **不要等用户说"跑测试"** — 改完代码自动触发
2. **不要一次改太多** — 每次只改一个功能点，跑测试，通过后再改下一个
3. **失败后不要慌** — 读错误消息，分析根因，针对性修复
4. **三次失败后必须求助** — 不要无限循环
5. **优先使用精确匹配的测试文件** — 不要一个命令跑全量测试（除非项目没有精确匹配）
6. **如果项目无测试** — 用 `run_shell` 至少跑一次构建/编译验证代码可运行

## 与浏览器工具配合

如果项目是前端且 `browser_tools` 可用（playwright 已安装）：

```
1. 修改代码后先跑单元测试
2. 单元测试通过后，启动 dev server → browser_navigate → 截图验证视觉
3. 如有 E2E 测试，同时运行 browser_navigate 检查关键页面
```
