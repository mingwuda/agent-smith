# 我用 3 天撸了一个桌面 AI 智能体，能写代码、管文件、跑脚本，完全开源免费

> 你有没有想过，如果 AI 不只是聊天，而是能直接操作你的电脑——帮你整理文件、执行脚本、生成日报——那该多爽？
>
> 我花了 3 天，把这个想法变成了现实。

---

## 一、为什么我要做这个东西？

事情是这样的。

上个月我加班到凌晨两点，不是因为在写什么牛逼的代码，而是在干这些破事：

- 整理桌面上 200 多个散落的文件，按项目归类
- 把 5 个 Excel 的数据汇总到一个报表里
- 写今天的周报、明天的日报、后天的会议纪要
- 跑一堆 Python 脚本处理数据，每个都要手动改参数

这些事难吗？不难。但烦吗？烦得要死。

我当时就在想：**我明明有 AI 啊，为什么这些事还得自己干？**

试了一圈市面上的工具：

- **ChatGPT / DeepSeek 网页版**：能聊天，但碰不了我的文件系统，给它个文件路径它都读不了
- **各种 AI IDE 插件**：写代码还行，但让它帮我整理桌面文件？它会说"这个超出我的能力范围了"
- **商业桌面智能体**：功能倒是全，但数据全传到别人服务器上，公司内网的代码和文档我敢传吗？

都不是我想要的。

**我想要的是：一个跑在自己电脑上的 AI，能看懂我的指令，能操作我的文件，能执行代码，数据不出本地，想用什么模型就用什么模型。**

于是我就自己写了一个。

## 二、这东西到底能干啥？

先放几个真实使用场景，你们感受一下。

### 场景 1：一句话生成日报

我在 SKILL.md 里定义了一个「日报生成」技能：

```markdown
# 日报生成助手
## Trigger
写日报、今日总结、生成日报

## Instructions
1. 询问用户今天完成了哪些工作
2. 按项目分类整理
3. 生成 Markdown 格式的日报文件
4. 保存到工作区的 reports/ 目录
```

然后我只需要对它说一句：**"帮我写今天的日报"**。

它就会问我今天干了啥，我随便说几句，它自动整理成结构化日报，保存到 `reports/daily-report-2026-05-24.md`。

### 场景 2：批量文件整理

> 我："把工作区里所有的 Python 文件找出来，按修改日期分类放到不同文件夹里"

它会：
1. 调用 `search_files` 搜索所有 `*.py` 文件
2. 调用 `run_python` 执行一段脚本，按日期分组
3. 调用 `write_file` 创建目录结构
4. 逐个移动文件

全程我就说了一句话，剩下的全自动。

### 场景 3：数据分析 + 报告

> 我："读取 data.csv，分析每月销售趋势，生成可视化图表"

它会：
1. 用 `read_file` 读取 CSV
2. 用 `run_python` 执行 pandas + matplotlib 分析
3. 输出图表文件和分析结论

不用打开 Jupyter，不用写一行代码，对话框里说句话就行。

## 三、技术架构：为什么选 LangGraph？

这个项目的核心是一个 **LangGraph ReAct Agent**。简单解释一下这个选择：

### 什么是 ReAct？

ReAct = **Re**asoning + **Act**ing。

传统的 AI 对话是「一问一答」——你问一个问题，它给一个答案。

ReAct 不一样。它会：

```
用户: 帮我整理桌面上的文件

Agent 思考: 我需要先看看桌面上有什么文件
  → 调用 list_files 工具
  → 得到文件列表

Agent 思考: 有很多 .py 和 .md 文件，我按类型分类
  → 调用 run_python 执行分类脚本
  → 得到分类结果

Agent 思考: 现在创建目录并移动文件
  → 调用 write_file 创建目录
  → 逐个移动

Agent: 整理完成！已将 47 个文件按类型分为 5 个目录。
```

**它不是一次性回答，而是一步一步思考、执行、观察、再思考。** 像一个真正的人在干活一样。

### 为什么选 LangGraph 而不是直接用 LangChain？

| 特性 | LangChain Agent | LangGraph ReAct |
|------|----------------|-----------------|
| 状态管理 | 无状态 | 有状态图，支持复杂流程 |
| 工具调用 | 基础支持 | 原生支持，错误可恢复 |
| 流式输出 | 不支持 | 支持 SSE 流式事件 |
| 会话持久化 | 需手动实现 | 内置 MemorySaver |
| 扩展性 | 一般 | 图节点可自由组合 |

LangGraph 本质上是一个**状态机框架**，每个节点是一个处理步骤，边是状态转移。这让 Agent 的执行流程非常可控，出了问题也能精确定位。

### 整体架构

## 四、核心功能拆解

### 1. 工具系统：让 AI 能「动手」

光能聊天的 AI 是废物。能干活的 AI 才有价值。

我给 Agent 注册了 9 个核心工具：

| 工具 | 功能 | 典型场景 |
|------|------|---------|
| `read_file` | 读取文件内容 | "帮我看看这个配置文件写了啥" |
| `write_file` | 写入文件 | "把这段代码保存到 test.py" |
| `append_to_file` | 追加内容 | "把今天的记录追加到日志里" |
| `list_files` | 列出目录 | "看看工作区里有什么" |
| `delete_file` | 删除文件 | "把临时文件删掉" |
| `search_files` | 搜索文件 | "找所有 .json 文件" |
| `run_python` | 执行 Python | "算一下这组数据的均值" |
| `get_system_info` | 系统信息 | "看看磁盘还剩多少空间" |
| `get_workspace_path` | 工作区路径 | "我的文件在哪" |

这些工具覆盖了日常 80% 的自动化需求。而且因为是 `@tool` 装饰器模式，加新工具只需要写一个函数：

```python
@tool
def my_new_tool(param: str) -> str:
    """工具描述，AI 会根据这个决定何时调用"""
    # 你的逻辑
    return "结果"
```

一行注册，Agent 就多了一种能力。

### 2. Skills 技能系统：Prompt 即插件

这是我觉得最有意思的设计。

传统的 AI 插件系统，你得写代码、注册 API、处理回调。太重了。

我的 Skills 系统只需要一个 **Markdown 文件**：

```markdown
# 代码审查助手

## Description
对代码进行审查，指出潜在问题

## Trigger
代码审查、review、检查代码

## Instructions
1. 读取用户指定的代码文件
2. 从以下维度审查：
   - 安全性：是否有注入、越权等风险
   - 性能：是否有明显性能问题
   - 可读性：命名、注释、结构
3. 输出结构化审查报告
4. 如有问题，提供修改建议

## Tools Required
read_file、write_file
```

把它放到 `samples/` 目录下，调用 `/skills/reload` 接口，**热加载，不用重启**。

这意味着什么？

**任何人都可以创建和分享技能。** 你写一个「周报生成」技能，我写一个「日志分析」技能，大家放到 Git 仓库里，一行命令就能装上。

### 3. 流式输出：所见即所得

大部分 AI 应用的问题是：你发一条消息，然后盯着空白屏幕等 10 秒。用户体验极差。

我用了 **SSE（Server-Sent Events）** 实现流式输出：

```
← token: "我" 
← token: "来" 
← token: "帮" 
← token: "你" 
← token: "看看" 
← tool_start: read_file {path: "data.csv"}
← tool_result: "文件内容..."
← token: "文件共有 1523 行数据..."
```

前端实时渲染，AI 每说一个字你都能看到，工具执行过程也完全透明。**不是等半天出一个结果，而是边想边说边干。**

### 4. 多会话管理

左侧栏支持多个会话并行：

- 「日常开发」会话：处理代码相关任务
- 「文档整理」会话：处理文件和报告
- 「数据分析」会话：专门跑数据脚本

每个会话独立的上下文，互不干扰，随时切换。会话数据持久化到 JSON 文件，重启不丢失。

### 5. 用量追踪

Token 花了多少、调用了几次、哪个模型用得最多——全部记录在案。

```
📊 今日统计
├── 总调用: 47 次
├── 输入 Token: 128,456
├── 输出 Token: 34,892
└── 估算费用: $0.23
```

## 五、怎么用？5 分钟上手

### 环境要求

- Python 3.10+（推荐 3.13）
- 一个 LLM API Key（DeepSeek / OpenAI / 通义千问 都行）

### 安装

```bash
# 克隆项目
git clone https://gitee.com/mingwuda/desktop-agent.git
cd desktop-agent

# 安装依赖（只有 5 个包，秒装）
pip install -r requirements.txt

# 启动
cd agent_core
python main.py
```

浏览器打开 `http://127.0.0.1:8899/`，点右上角设置，填入 API Key，选模型，开聊。

### 配置模型

支持任何兼容 OpenAI 接口的模型：

| 模型 | API 地址 | 价格 |
|------|---------|------|
| DeepSeek Chat | `https://api.deepseek.com` | 极便宜 |
| GPT-4o | 留空（默认 OpenAI） | $2.5/M tokens |
| 通义千问 | `https://dashscope.aliyuncs.com/compatible-mode` | 有免费额度 |
| 任意 OpenAI 兼容 | 填对应地址 | 看提供商 |

## 六、为什么不用现成的商业产品？

说实话，市面上不缺 AI 工具。ChatGPT、Claude、Copilot、各种 Agent 框架……为什么还要自己造？

**因为三个字：可控性。**

| 维度 | 商业产品 | 本项目 |
|------|---------|--------|
| 数据隐私 | 上传云端 | **100% 本地** |
| 模型选择 | 绑定一家 | **任意切换** |
| 功能扩展 | 等官方更新 | **自己加，热加载** |
| 费用 | 订阅制 $20-200/月 | **只付 API 调用费** |
| 部署方式 | 依赖他人服务器 | **跑在自己机器上** |
| 代码透明 | 黑盒 | **完全开源** |

特别是数据隐私这一点。如果你是开发者，你的代码仓库、API Key、内网文档——这些能传到第三方服务器上吗？

**自己的数据，就应该跑在自己的机器上。**

## 七、和 WorkBuddy、Codex 是什么关系？

如果你关注 AI Agent 领域，一定听过这两个名字：

- **OpenAI Codex**：编程 Agent，擅长写代码、修 Bug、跑测试
- **腾讯 WorkBuddy**：职场助手，擅长写文档、做 PPT、整理文件

这个 Desktop Agent 和它们的区别在于：

**Codex 是「程序员的 AI」，WorkBuddy 是「白领的 AI」，而这个项目是「所有人的 AI 工具箱」。**

它没有 Codex 那么强的编码能力（毕竟底层模型可以自由选择），也没有 WorkBuddy 那么深的腾讯生态集成。但它有：

- ✅ **完全开源**，代码你随便改
- ✅ **完全本地**，数据你做主
- ✅ **完全自由**，模型你来选
- ✅ **完全可扩展**，Skills 你来写

**它不是一个产品，它是一个平台。** 你可以在上面搭建自己的 AI 工作流。
